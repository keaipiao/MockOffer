## 2026-06-16 · Spring Boot 4 接 Testcontainers：BOM 要自己导、2.0 的 artifactId 加前缀、测试 starter 不含 TestRestTemplate

- 现象：① pom 加 `org.testcontainers:junit-jupiter` / `postgresql` 报 `version missing`（整个 POM 都读不了）；② 导了 testcontainers-bom 后仍 version missing；③ 编译报 `org.springframework.boot.test.web.client` 包不存在、找不到 `TestRestTemplate`。
- 根因：(a) Boot 4.1 的 dependencies BOM 不再直接管 testcontainers 子模块版本，要在 dependencyManagement 显式 import `testcontainers-bom`；(b) testcontainers 2.0 把所有 artifactId 统一加 `testcontainers-` 前缀（`junit-jupiter`→`testcontainers-junit-jupiter`、`postgresql`→`testcontainers-postgresql`）；(c) Boot 4 模块化测试 starter（`spring-boot-starter-webmvc-test`）不含 `TestRestTemplate`。
- 正确做法：① 显式 import testcontainers-bom，版本去查 `spring-boot-testcontainers:<boot版本>` 的 POM 依赖的那条对齐（Boot 4.1.0 → 2.0.5）；② artifactId 用 2.0 的 `testcontainers-*` 前缀名；③ 集成测试 HTTP 客户端改用 spring-web 的 `RestClient`（`.exchange(fn,false)` 不抛 4xx）、端口用 `@Value("${local.server.port}")`、容器连接用 `@DynamicPropertySource`（避开 Boot4 包路径变动）。
- 适用范围：Spring Boot 4.x 项目加 Testcontainers 集成测试、或用 testcontainers 2.x；后续里程碑写集成测试照此配。

## 2026-06-15 · Spring Boot 4 模块化：第三方集成需配套 starter，否则 autoconfig/方言缺失

- 现象：① 用 `com.fasterxml.jackson.databind` 编译报找不到包；② 仅加裸 `flyway-core` 时 Flyway 不执行迁移、JPA `ddl-auto=validate` 直接报缺表；③ 改用 `spring-boot-starter-flyway` 后又报 `Unsupported Database: PostgreSQL 18.x`。
- 根因：Spring Boot 4 把单体 autoconfigure 拆成模块化，并默认升级到 Jackson 3。(a) Jackson 3 包名迁到 `tools.jackson.databind`（注解仍 `com.fasterxml.jackson.annotation` 不变）；(b) Flyway 不再靠裸 `flyway-core` 自动配置，必须用 `spring-boot-starter-flyway` 才有 autoconfig；(c) starter 只带 flyway-core，数据库方言要单独加 `flyway-database-postgresql`。
- 正确做法：Spring Boot 4 下第三方集成优先找对应 `spring-boot-starter-*`；Jackson 的 import 用 `tools.jackson.*`；数据库方言模块单独加。
- 适用范围：Spring Boot 4.x 项目加任何需 autoconfig 的依赖（Flyway / 各数据库方言 / JSON 等）；后续里程碑加依赖照此排查。

## 2026-06-14 · PostgreSQL 18+ Docker 镜像数据卷路径变更

- 现象：docker compose 起 `postgres:18-alpine` 容器立即 Exited(1)，日志「in 18+, these Docker images are configured to store database data in a 子目录…」，依赖它（`condition: service_healthy`）的后端连带起不来。
- 根因：PostgreSQL 18 官方镜像把 PGDATA 移到 `/var/lib/postgresql` 下的子目录、VOLUME 声明从 `/var/lib/postgresql/data` 改为 `/var/lib/postgresql`；仍按旧习惯把具名卷挂到 `/var/lib/postgresql/data` 会被镜像主动拒绝启动（防数据混淆/丢失）。
- 正确做法：`postgres:18+` 的数据卷挂到 `/var/lib/postgresql`（不是 `/data`）。
- 适用范围：任何用 postgres:18 及以上官方镜像 + 具名卷的 compose / k8s 配置。
