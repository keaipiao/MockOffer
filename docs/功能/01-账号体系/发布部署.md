## 2. 改动汇总

### 2.1 代码改动

- **后端**：邮箱验证码无密码登录、GitHub OAuth 登录（登录/绑定分流）、当前用户/登出/刷新、登录方式绑定·解绑·换绑邮箱；JWT 存 httpOnly cookie 的无状态鉴权、jti 黑名单即时吊销、refresh 轮换。
- **前端**：登录页（邮箱两步 + GitHub 入口）、账号设置（登录方式管理）、首页（未登录落地页 / 登录后工作台）、全站顶部导航；鉴权改 **SSR + AuthProvider**（根 layout 服务端读 cookie 解析用户，零闪烁）。

### 2.2 数据变更 / 迁移

Flyway `V1__create_account.sql`：建 `users`（自增 id）与 `user_identities`（雪花 id，含 `(provider, provider_uid)` 软删部分唯一索引）。容器启动时自动执行。

### 2.3 配置 / 基础设施变更

- 新增 **Redis**（验证码存储 + 限流 + 会话黑名单/refresh）。
- 新增 **GitHub OAuth App**（见下）。
- 前端容器新增 SSR 用的后端内网地址变量 `BACKEND_INTERNAL_URL`。

**GitHub OAuth App 申请**（GitHub → Settings → Developer settings → OAuth Apps → New OAuth App）：
- Homepage URL：`https://mockoffer.com`（dev：`http://localhost:3000`）
- **Authorization callback URL**：填**后端**地址 + `/login/oauth2/code/github`
  - dev：`http://localhost:8080/login/oauth2/code/github`
  - prod：`https://<后端公网域名>/login/oauth2/code/github`
- 创建后拿 Client ID、生成 Client Secret，填进 `deploy/.env` 的 `GITHUB_CLIENT_ID` / `GITHUB_CLIENT_SECRET`。

### 2.4 环境变量

| 变量名 | 默认值 | 说明 | 是否必填 |
|---|---|---|---|
| `POSTGRES_USER` / `POSTGRES_PASSWORD` / `POSTGRES_DB` | mockoffer | PostgreSQL 凭证与库名 | 是（prod 改强密码） |
| `JWT_SECRET` | dev 占位串 | JWT HS256 密钥，≥32 字节 | 是（prod 必换） |
| `GITHUB_CLIENT_ID` / `GITHUB_CLIENT_SECRET` | dev-placeholder | GitHub OAuth 凭证 | 用 GitHub 登录则必填 |
| `TENCENT_SES_SECRET_ID` / `TENCENT_SES_SECRET_KEY` | 空 | 腾讯云 SES 凭证 | prod 真实发码必填 |
| `SES_FROM_EMAIL` / `SES_TEMPLATE_ID` | 空 | 发件邮箱与审核通过的模板 ID | prod 真实发码必填 |
| `SPRING_PROFILES_ACTIVE` | dev | 环境 profile，compose 设 `prod` | 是（prod） |
| `BACKEND_INTERNAL_URL` | http://localhost:8080 | 前端 SSR server→后端 内网地址；compose 设 `http://backend:8080` | 容器部署必填 |
| `SPRINGDOTENV_IGNORE_IF_MISSING` | — | 容器内无 `.env` 时不报错，compose 设 `true` | 容器部署必填 |

> 新增 env 须**同步到所有相关位置**（`deploy/.env.example`、`deploy/docker-compose.yml`、本文件、`docs/部署手册.md`），漏一处线上就会起不来。

## 3. 发布步骤（本次上线）

1. 准备：拉取新代码；`cp deploy/.env.example deploy/.env` 并填真实值（DB 密码、`JWT_SECRET`、GitHub 凭证、SES 凭证）。
2. 数据迁移：无需手动——后端启动时 Flyway 自动执行 `V1`。
3. 构建并部署：`cd deploy && docker compose up -d --build`（PG / Redis / MinIO / 后端 / 前端）。
4. **冒烟测试**：
   - `curl http://<后端>/api/health` → 200
   - 浏览器访问前端：未登录见落地页；邮箱验证码登录 / GitHub 登录跑通；设置页绑定解绑正常。

---

## 4. 回滚步骤

1. 代码回滚：`docker compose` 切回上一个稳定镜像/标签并 `up -d`。
2. 数据回滚：迁移只新增 `users` / `user_identities` 两表、不改旧数据；回退代码即可，新表留着不影响旧逻辑（设计阶段的回滚边界见 `设计.md` §3.8）。
3. 验证：回滚后再跑一遍 §3.4 冒烟测试。
